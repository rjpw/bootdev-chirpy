## Test Client Pattern (AI-contributed: Kiro)

After discussing the cost of multi-step workflow tests (token threading, fixture management, manual request construction), Kiro proposed a `testClient` abstraction — a stateful actor that models a user session against the API.

**Core idea:** Each test instantiates one `testClient` per actor. The client manages its own tokens, session IDs, and auth headers internally. Methods like `Login`, `Chirp`, `Refresh`, `Revoke` express intent; the plumbing is invisible.

**Before (current):**
- Global `testCache` shared across steps
- Manual token passing between cases
- Fixture files for every request variant
- Tests read as HTTP mechanics

**After (proposed):**
```go
saul := newTestClient(t, srv)
saul.CreateUser("saul@bettercall.com", "password")
saul.Login("saul@bettercall.com", "password")
assertStatus(t, saul.Chirp("Better call Saul!"), http.StatusCreated)
```

**Benefits:**
- State is per-actor, not global — multiple users in one test without collision
- Token management is automatic after login
- Scenarios read as narratives, not request/response plumbing
- Each new endpoint = one new method; scenarios stay cheap to write
- Fixture files become optional (inline params for simple cases)

**Where it lives:** `internal/httpapi/testclient_test.go` — test infrastructure for the adapter layer, built on existing helpers (`issueRequest`, `issueAuthorizedRequest`).

**Growth path:** Build it in the first week of a project, right after the first endpoint. It grows with the system. By the time auth workflows arrive, the vocabulary is already there.
