## Lesson Notes

With my refactoring in place, I can now decide if I want to implement the webhook lesson right in the Postgres implementation, i.e. working inside-out as the lessons tend to, or whether I want to stick with my plan of working outside-in as I'd prefer. My goal for this project (beyond the course requirements) was to build good habits, focusing on API first while imposing testable architectural boundaries.

This lesson introduces one general idea and two related ones from the world of [Enterprise Integration Patterns](https://www.enterpriseintegrationpatterns.com/):

**Webhooks** - The webhook idea is the one general idea, and it's pretty straight-forward. It's basically a server-to-server event publishing arrangement, where you ask a provider (Stripe in their example) to notify you after important things happen. Note: API keys show up in a later lesson.

**Events** - Event-driven architecture is the topic of a [different course](https://www.boot.dev/courses/learn-pub-sub-rabbitmq-golang), so I'm not surprised this is getting a light touch here. But I do have an issue with the name they gave this event. We're asked to react to `user.upgraded`, which feels misleading. It describes a milestone in the future (the lesson asks us to upgrade the user), not a fact from the past. I would have expected an event from `Stripe` like `user.payment_succeeded`. If there was someone downstream of this server, and we succeeded in upgrading our user, only then would I expect our server to POST a webhook event indicating that a `user.upgraded` event occurred. But I quibble.

**Idempotence** - The course explains very little about the value of idempotence. Having implemented a [Mojaloop](https://docs.mojaloop.io/legacy/) payment scheme adapter twice, first in [Vert.X](https://vertx.io/) then again in [Apache Camel](https://camel.apache.org/), I know it can turn HTTP into a system for supporting exactly-once delivery semantics in a fragile ecosystem. I think they could have mentioned holding onto a list of incomplete request flows (i.e. with some kind of correlation ID), and an operational window or TTL that lets us keep the clustered cache performant, but again I quibble.

Given what I know about how deep this problem can get, and noting that their example represents our server fulfilling a change of account status based on our receipt of payment, I'm going to have to work hard to resist making a meal of this little lessson. For now let's just get to the end of the course without breaking any self-imposed speed limits.
